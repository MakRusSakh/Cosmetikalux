## 2. Архитектура фронтенда

### Рендеринг

| Страница | Стратегия | Причина |
|----------|-----------|---------|
| Главная | SSG + ISR (60s) | Редко меняется, максимальная скорость |
| Каталог | SSR | Фильтры в URL, SEO, актуальные остатки |
| Товар | SSR | Актуальные остатки/цены из 1С |
| Корзина | CSR | Персональные данные, нет нужды в SEO |
| Чекаут | CSR | Персональные данные |
| ЛК | CSR | Авторизация, персональные данные |
| Блог | SSG + ISR (300s) | Статический контент |
| О нас, доставка | SSG | Полностью статические |
| Админка | CSR | Не нужно SEO |

## 3. Архитектура бэкенда

### API слои

```
┌──────────────────────────┐
│      Route Handler        │  ← Next.js API route
│  (валидация, auth check)  │
└───────────┬──────────────┘
            │
┌───────────▼──────────────┐
│      Service Layer        │  ← Бизнес-логика
│  (productService,         │
│   orderService,           │
│   chatService, etc.)      │
└───────────┬──────────────┘
            │
┌───────────▼──────────────┐
│      Data Layer           │  ← Prisma ORM
│  (prisma.product.find...) │
│  + Redis cache            │
└──────────────────────────┘
```

### Пример: создание заказа

```
POST /api/orders
       │
       ▼
1. Валидация данных (Zod)
2. Проверка авторизации (JWT)
3. Проверка наличия товаров (Redis → PostgreSQL)
4. Расчёт стоимости (товары + доставка - скидки - бонусы)
5. Создание заказа в PostgreSQL (транзакция)
6. Резервирование товаров (stock - reserved)
7. Создание платежа в ЮKassa
8. Возврат confirmation URL
       │
       ▼
Покупатель оплачивает на странице ЮKassa
       │
       ▼
Webhook от ЮKassa (payment.succeeded)
       │
       ▼
1. Проверка подписи webhook
2. Обновление статуса заказа → PAID
3. Начисление бонусов
4. Отправка заказа в 1С
5. Создание отправления в СДЭК/Почте
6. SMS + Email покупателю
7. Уведомление менеджеру
```

## 5. Архитектура синхронизации с 1С

### Модель «Push + Pull»

```
    ┌─────────┐                          ┌───────────┐
    │   1С    │                          │   Сайт    │
    │         │                          │           │
    │         │──── Push остатки ────────>│           │
    │         │──── Push цены ──────────>│           │
    │         │──── Push новые товары ──>│           │
    │         │──── Push статус заказа ─>│           │
    │         │                          │           │
    │         │<─── Pull новые заказы ───│           │
    │         │                          │           │
    └─────────┘                          └───────────┘
```

### Обработка конфликтов

| Ситуация | Решение |
|----------|---------|
| Товар куплен, но в 1С остаток 0 | Заказ создаётся, менеджер решает вручную |
| Цена изменилась во время оформления | Фиксируется цена на момент добавления в корзину |
| 1С недоступна | Redis кэш, уведомление админу, retry через 5 мин |
| Дубль заказа в 1С | Идемпотентность по orderNumber |

### Формат обмена

```
Вариант 1 (основной): REST API (JSON)
- Простой, быстрый, стандартный
- 1С выступает клиентом нашего API

Вариант 2 (альтернативный): CommerceML (XML)
- Если 1С не может вызывать HTTP
- Обмен через файлы в формате CommerceML
- Cron-задача проверяет папку обмена
```

## 7. Архитектура доставки

### Расчёт стоимости

```
Покупатель выбирает город
         │
         ▼
┌────────────────────────────┐
│  Параллельные запросы:      │
│                             │
│  СДЭК API → тариф ПВЗ     │
│  СДЭК API → тариф курьер  │
│  Почта API → тариф         │
│                             │
│  + Локальная доставка       │
│    (если Южно-Сахалинск)   │
│                             │
│  + Проверка бесплатной      │
│    доставки (от 5000₽)     │
└────────────────────────────┘
         │
         ▼
Покупатель видит все варианты с ценами
```

### Вес товаров

- Каждый товар имеет поле weight (граммы)
- Суммарный вес корзины → запрос к API доставки
- Габариты: стандартная коробка (размер зависит от количества)
- Формулы:
  - 1-3 товара: 20×15×10 см
  - 4-8 товаров: 30×20×15 см
  - 9+ товаров: 40×30×20 см

## 8. Безопасность — архитектура

```
┌─────────────────────────────────────────────────┐
│                  Cloudflare                       │
│  DDoS protection, WAF, rate limiting             │
├─────────────────────────────────────────────────┤
│                  Next.js Middleware               │
│  CORS, CSRF, auth check, rate limiting           │
├─────────────────────────────────────────────────┤
│                  API Layer                        │
│  Zod validation, sanitization, auth guards       │
├─────────────────────────────────────────────────┤
│                  Service Layer                    │
│  Business logic validation, permissions          │
├─────────────────────────────────────────────────┤
│                  Data Layer                       │
│  Prisma (parameterized queries), encryption      │
└─────────────────────────────────────────────────┘
```

### JWT Flow

```
Login → { accessToken (15min), refreshToken (30d, httpOnly cookie) }

Каждый запрос:
1. Middleware проверяет accessToken в Authorization header
2. Если истёк → автоматический refresh через refreshToken cookie
3. Если refresh истёк → logout, редирект на /auth/login
```

## 9. Деплой и инфраструктура

### Вариант A: Vercel (рекомендуемый)

```
GitHub → Vercel (автодеплой)
PostgreSQL → Neon / Supabase
Redis → Upstash
Files → Vercel Blob / S3-совместимое хранилище
```

**Плюсы:** нулевой DevOps, автоскейлинг, edge-функции, preview deploys.
**Минусы:** стоимость при росте, привязка к платформе.

### Вариант B: VPS (бюджетный)

```
GitHub Actions → Docker → VPS (Timeweb / Selectel)
PostgreSQL → на VPS
Redis → на VPS
Nginx → reverse proxy + SSL
Files → на VPS или S3
```

**Плюсы:** полный контроль, предсказуемая стоимость.
**Минусы:** требует DevOps, ручное масштабирование.

### Docker Compose (для VPS)

```yaml
services:
  app:
    build: .
    ports: ["3000:3000"]
    env_file: .env
    depends_on: [postgres, redis]

  postgres:
    image: postgres:16
    volumes: ["pg_data:/var/lib/postgresql/data"]
    environment:
      POSTGRES_DB: cosmetikalux
      POSTGRES_USER: clx
      POSTGRES_PASSWORD: ${DB_PASSWORD}

  redis:
    image: redis:7-alpine
    volumes: ["redis_data:/data"]

  nginx:
    image: nginx:alpine
    ports: ["80:80", "443:443"]
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./certs:/etc/nginx/certs
```
